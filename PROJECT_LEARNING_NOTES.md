## 0. 30 秒项目介绍

**人话版**

这个项目解决的是本地生活门店前台工作太依赖人工的问题。用户可能一句话里同时问服务、约时间、指定技师、表达偏好，人工前台要反复确认、查档期、记录偏好、处理改约取消，还容易漏信息。这个系统就是把这些工作变成一个可追踪、可确认、可测试的 Agent 流程。

**专业回答**

项目面向本地生活服务门店中咨询、预约、偏好记录、排班校验依赖人工前台的问题，构建了一个业务型 Agent 系统。系统基于单一 Operations Agent（`OperationsAgent`）编排层和 LangGraph 状态流，将用户自然语言需求转化为服务咨询、预约槽位管理、客户记忆沉淀、排班/技师校验、写操作确认、异常兜底和 trace 记录等可执行流程。重点不是做聊天机器人，而是把门店前台的业务闭环做成可治理、可观测、可回归测试的 Agent runtime。

**一句话背诵**

我做的是一个面向本地生活门店的业务型 Agent，把自然语言里的咨询、预约、偏好、档期校验和确认写入，编排成可追踪、可测试、可兜底的前台运营流程。

## 1. 面试官最可能先问的问题

### Q1：这个项目到底解决了什么问题？

**人话解释**

它不是单纯“帮用户预约”。真正的问题是：前台工作很碎，用户说话也不结构化。人工要理解用户意思、追问缺失信息、查技师和档期、确认后才能写入，还要记住用户偏好。这个项目把这些碎工作串成一个稳定流程。

**专业回答**

项目解决的是本地生活门店前台运营中的多步骤协同问题。用户输入通常是自然语言，里面可能包含服务咨询、预约意图、时间信息、技师偏好、特殊要求或取消/改约需求。系统通过 `OperationsAgent` 将这些输入转成结构化状态，按流程完成意图识别、槽位补齐、工具规划、排班校验、用户确认、写操作执行、异常兜底和 trace 记录，减少人工前台反复追问、漏记偏好和排班冲突的风险。

**追问怎么答**

如果面试官问“为什么不用普通表单”，可以说：表单适合用户主动填写完整信息，但真实门店对话经常是缺字段、多轮补充、临时改时间、指定技师、混合咨询和预约。Agent 的价值在于把自然语言对话转换成结构化业务流程。

### Q2：为什么说它是业务型 Agent，不是普通 chatbot？

**人话解释**

普通 chatbot 主要是回答话。这个项目的重点是“做事”：查档期、判断信息是否完整、生成确认请求、写预约、写记忆、触发人工兜底，而且每一步都有规则和审计。

**专业回答**

它不是以自由对话为中心，而是以业务状态推进为中心。系统会维护 `booking_slots`、`missing_slots`、`tool_plan`、`tool_results`、`confirmation_request`、`customer_context` 和 `trace_events` 等状态。LLM 或规则层负责理解和编排，真正的业务动作必须通过受控工具执行。创建预约、改约、取消和客户记忆写入都需要 Tool Gateway 校验 schema、权限和 confirmation token，避免模型绕过业务规则直接写库。

**一句话背诵**

这个项目的核心不是让模型“会聊天”，而是让模型在明确业务约束下推进一个可验证的运营流程。

### Q3：为什么选择单一 OperationsAgent，而不是多个 Agent 分别负责咨询、预约、记忆？

**人话解释**

因为这些任务在真实对话里经常混在一起。用户可能先问项目，再说“那我明天下午约一下，还要安静房间”。如果拆很多 Agent，各自维护状态，容易丢上下文。

**专业回答**

我选择单一 `OperationsAgent` 编排层，是因为门店前台场景里的咨询、预约、改约、偏好和异常反馈强耦合，核心难点是同一段多轮对话里的状态一致性。如果拆成多个自治 Agent，容易出现状态分散、上下文回传复杂、重复追问和工具权限边界不清的问题。因此我把编排收敛到一个 OperationsAgent，用 LangGraph 显式管理节点流程；将确定性动作下沉为 Tools，例如档期查询、技师查询、预约写入、知识库检索和客户记忆写入。这样既保留 Agent 的自然语言理解能力，也让业务动作可治理。

**追问怎么答**

如果问“那多 Agent 有没有价值”，可以回答：有，但不适合这个阶段作为主编排。后续可以把知识检索、排班优化、运营分析做成更独立的子能力，但当前最重要的是保证预约状态、确认写入和 trace 链路稳定。

### Q4：为什么选 LangGraph？

**人话解释**

因为这个流程不是一次问答，而是一个有步骤、有状态、有分支的流程。LangGraph 比简单链式调用更适合把每一步显式建出来。

**专业回答**

LangGraph 适合状态化、多节点、可追踪的 Agent 编排。这个项目需要把“识别需求 - 补齐信息 - 查档期 - 生成确认 - 执行工具 - 生成回复 - 记录 trace”建模成可观察流程，而不是把所有逻辑塞进一个 prompt。使用 LangGraph 的好处是节点边界清晰，每个节点只处理一类状态更新，方便测试、调试和 trace replay，也方便在工具失败、信息缺失、敏感场景时插入兜底逻辑。

**专业表达模板**

我选 LangGraph 不是为了追框架，而是因为这个业务天然是状态机：每一轮对话都要读取上一轮状态、更新槽位、判断缺失字段、决定工具调用和确认策略。LangGraph 能把这些步骤显式化，降低黑盒 prompt 编排的风险。

### Q5：项目里 LangChain 和 LangGraph 的关系是什么？

**人话解释**

LangGraph 是流程骨架，LangChain 更多是生态和组件。不要把 LangChain 说成主编排核心。

**专业回答**

当前项目的运行时核心是 LangGraph。LangGraph 负责将 OperationsAgent 的状态推进建模为节点流程；LangChain 相关能力更多用于模型、prompt 或生态兼容。面试时我会强调 LangGraph 是主编排层，因为它承载了状态机、节点流转和可观测性设计。

## 2. Agent 编排设计

### Q6：OperationsAgent 一轮对话大概怎么跑？

**人话解释**

用户发一句话后，系统先初始化本轮状态，再判断用户要干什么，然后加载客户记忆，抽取预约信息，判断是否要写记忆或查知识库，再规划工具，最后生成回复并记录 trace。

**专业回答**

一轮 operations turn 可以概括为：初始化 state，识别 intent，加载 customer context，提取或合并 booking slots，生成 memory proposal，规划 tool calls，执行工具或生成 confirmation request，生成最终回复，做输出策略检查，最后写入 trace。这样每一步都有明确输入输出，不依赖模型在一个长 prompt 里隐式记忆流程。

**可背流程**

```text
initialize_turn
  -> classify_intent
  -> load_customer_context
  -> extract_booking_slots
  -> propose_memory_writes
  -> plan_tool_calls
  -> execute_tools
  -> generate_response
  -> output_policy_check
  -> finalize_turn
```

### Q7：为什么要显式建模这些节点？

**人话解释**

为了知道每一步发生了什么。否则出了问题只能猜是模型没理解、槽位没抽到、工具没调，还是确认没过。

**专业回答**

显式节点建模可以把 Agent 行为拆成可测试、可观测的工程单元。例如意图识别错了，就看 `classify_intent`；预约字段缺失，就看 `extract_booking_slots`；工具没执行，就看 `plan_tool_calls` 或 `execute_tools`；回答不合规，就看 `output_policy_check`。这种设计让 Agent 从“黑盒聊天”变成“可定位的业务流程”。

### Q8：这个 Agent 的状态里最重要的字段有哪些？

**人话解释**

最重要的是：用户是谁、这轮说了什么、当前想做什么、预约信息填了哪些、缺什么、准备调什么工具、工具结果是什么、要不要确认、最后怎么回复、trace 怎么记录。

**专业回答**

核心 state 包括 `user_id`、`conversation_id`、`message`、`intent`、`booking_slots`、`booking_slot_sources`、`missing_slots`、`customer_context`、`memory_proposals`、`tool_plan`、`tool_results`、`confirmation_required`、`confirmation_request`、`reply`、`trace_id`、`trace_events` 和 `errors`。这些字段覆盖了多轮对话状态、业务动作规划、确认拦截、可观测性和异常处理。

### Q9：单一 Agent 会不会太复杂？

**人话解释**

会有复杂度，但复杂度被拆在节点和工具里了，不是堆在一个函数或 prompt 里。

**专业回答**

单一 OperationsAgent 不是把所有逻辑混在一起，而是统一维护业务状态，再通过 LangGraph 节点和 Tool Gateway 拆分职责。编排层负责“下一步做什么”，工具层负责“怎么可靠执行”。这种设计比多个 Agent 各自持有状态更容易保证一致性，也更容易做测试和 trace replay。

## 3. 预约状态管理

### Q10：booking_slots 是什么？为什么重要？

**人话解释**

它就是一张还没最终提交的预约草稿。用户说一句，系统就往草稿里填一点。缺什么就追问什么，完整了才去查档期和确认写入。

**专业回答**

`booking_slots` 是预约流程中的结构化状态，包含服务项目、日期、时间段、时长、指定技师、特殊偏好和客户标识等字段。自然语言对话是不稳定的，用户可能分多轮补充信息，所以系统需要把每一轮提取到的信息增量合并进 booking_slots。只有关键字段完整后，才进入排班校验和写操作确认。

### Q11：怎么支持多轮补充？

**人话解释**

第一轮用户可能只说“我想约肩颈”，系统发现缺日期和时间，就追问。第二轮用户说“明天下午三点”，系统把新信息补进原来的预约草稿，而不是重新开始。

**专业回答**

多轮补充通过保留并回传 `booking_slots` 实现。每一轮先读取已有槽位，再从当前 message 中提取增量字段，最后合并成新的状态。字段不完整时更新 `missing_slots`，生成追问；字段完整时才规划 `find_available_staff`、`check_schedule` 和写操作确认。这样可以降低重复追问和上下文丢失。

### Q12：用户临时改时间或换技师怎么办？

**人话解释**

把用户新说的字段覆盖到旧草稿里，不把整个预约重置。比如原来是下午三点，用户说“四点可以吗”，就只更新时间字段。

**专业回答**

状态更新采用增量覆盖策略。系统维护 `booking_slot_sources` 记录字段来源，例如用户本轮输入、上一轮状态、客户记忆或系统默认。用户修改时间、服务、技师或特殊偏好时，只覆盖对应字段，并保留其他已确认字段。这样能支持自然对话里的临时调整，同时保持预约状态连续。

### Q13：为什么要记录字段来源 booking_slot_sources？

**人话解释**

为了知道某个字段是用户刚说的、之前说的、系统补的，还是客户记忆带来的。后面出问题可以解释。

**专业回答**

字段来源用于可解释性和风险控制。比如 `special_requests` 可能来自用户本轮输入，也可能来自客户历史偏好；`customer_name` 可能是系统补齐。记录来源后，确认摘要可以更清楚，也便于 trace 和评测判断系统是否误用了记忆或覆盖了用户本轮输入。

### Q14：什么时候进入写入确认？

**人话解释**

只有预约关键信息完整，并且读工具校验过可用性后，才会生成“是否确认创建预约”的请求。

**专业回答**

系统只有在关键槽位完整、没有 `missing_slots`，并且完成必要的读工具校验后，才会规划写工具，例如 `create_booking`、`reschedule_booking` 或 `cancel_booking`。这些写工具不会直接执行，而是由 Tool Gateway 返回 `confirmation_required` 和 `confirmation_request`，等待用户确认后再执行。

## 4. 工具调用治理

### Q15：为什么要做 Tool Gateway？

**人话解释**

因为不能让 LLM 想写库就写库。工具网关就像门禁：参数对不对、权限够不够、用户确认没确认，都要先检查。

**专业回答**

Tool Gateway 是 Agent 和真实业务动作之间的治理边界。所有业务工具都声明 Pydantic 输入/输出 schema、permission 和 `requires_confirmation`。Gateway 对未知工具、参数错误、未确认写操作、handler 异常和超时返回结构化错误，并记录 trace event。这样可以把 LLM 的规划能力和后端的确定性执行隔离开，避免模型绕过业务规则直接写库。

### Q16：哪些工具需要确认？

**人话解释**

会改变业务状态或客户数据的都要确认，比如创建预约、改约、取消、写客户记忆、删除客户记忆。

**专业回答**

写操作和敏感操作需要确认，包括 `create_booking`、`reschedule_booking`、`cancel_booking`、`write_customer_preference` 和 `delete_customer_memory`。读工具如服务查询、技师查询、档期校验、知识库检索、客户资料读取可以直接执行，但也会记录 trace。

### Q17：confirmation token 的作用是什么？

**人话解释**

防止用户上一轮确认的是 A 操作，下一轮却被系统拿去执行 B 操作。

**专业回答**

confirmation token 用来绑定 conversation、tool name 和 tool arguments。用户确认时，下一轮请求必须带回相同工具名、参数和 token，Gateway 才允许执行写工具。这能防止确认请求被串用、参数被悄悄替换或模型伪造已确认状态。

### Q18：为什么还要用 Pydantic schema？

**人话解释**

因为自然语言提取出来的参数可能不完整或格式不对，schema 可以在调用工具前拦住。

**专业回答**

Pydantic schema 是工具调用的输入契约。它能保证工具 handler 收到的是结构化、类型正确、字段完整的数据。如果参数非法，Gateway 返回 validation error，不会调用 handler。这样能把模型输出的不确定性限制在工具边界外。

### Q19：工具调用失败怎么办？

**人话解释**

不能假装成功。读工具可以有限重试，写工具不能随便重试，避免重复创建预约。

**专业回答**

读工具可以设置有限 retry，因为读操作通常没有副作用；写工具和敏感工具不重试，避免重复写入或状态污染。工具异常会被 Gateway 转成结构化错误，并清洗内部异常文本，避免暴露敏感信息。若连续关键工具失败或排班状态不确定，Agent 会触发人工兜底。

## 5. 档期冲突处理

### Q20：怎么判断档期冲突？

**人话解释**

不是只看开始时间是不是一样，而是看两个时间段有没有重叠。

**专业回答**

档期冲突基于时间区间重叠判断。新预约的开始时间早于已有预约结束时间，并且新预约结束时间晚于已有预约开始时间，就说明两个时间段重叠。伪公式是：

```text
new_start < existing_end AND new_end > existing_start
```

这种判断比只比较开始时间更可靠，能覆盖 15:00-16:00 和 15:30-16:30 这类重叠情况。

### Q21：为什么创建预约前还要二次校验？

**人话解释**

因为用户确认前后，档期可能已经被别人占了。确认不等于一定能写入。

**专业回答**

排班是动态资源。即使在生成确认请求前已经查过可用性，用户确认时也可能发生并发变化。因此在真正执行创建或改约前，需要再次校验技师可用性和 busy schedule。若发现冲突，系统应阻断写入，并返回替代技师、临近时间或人工兜底建议。

### Q22：指定技师不可用怎么处理？

**人话解释**

不能强行创建，也不能随便换人。要告诉用户指定技师不可用，并给可选替代方案。

**专业回答**

当用户指定技师不可用时，系统会阻断写工具执行，生成 `booking_issue`，并返回可用替代技师或临近时间建议。这样既尊重用户偏好，又避免错误落库。若工具结果不确定或连续失败，则升级人工处理。

## 6. 客户记忆策略

### Q23：为什么要做客户记忆？

**人话解释**

门店服务很依赖偏好，比如喜欢安静房间、不喜欢大力度、对精油过敏、不想接收营销。记住这些可以减少重复沟通，但也必须谨慎。

**专业回答**

客户记忆用于沉淀长期偏好和服务约束，帮助后续预约更个性化。但记忆具有风险：可能过期、冲突、敏感或用户要求删除。因此系统采用 MemoryProposal 机制，先提取候选记忆，再经过确认、审核、去重、冲突处理和生命周期管理，避免直接把模型理解写成长期事实。

### Q24：MemoryProposal 是什么？

**人话解释**

它是“候选记忆”，不是正式记忆。先把用户说的话整理成结构化建议，再决定要不要写、怎么写。

**专业回答**

`MemoryProposal` 包含 type、content、evidence、confidence、sensitivity、requires_confirmation 和可选 expires_at。规则和 LLM 都只能产出 proposal，不能直接写库。真正写入必须通过 `write_customer_preference` 工具，并经过 Tool Gateway 确认。

### Q25：普通偏好和敏感记忆怎么区分？

**人话解释**

“喜欢安静房间”这种是普通偏好；“精油过敏”“不要营销”这种会影响安全、隐私或合规，要更谨慎。

**专业回答**

普通偏好如房间氛围、力度偏好、沟通偏好，经确认后可以成为 active memory，并用于后续预约建议。敏感类型如 `service_contraindication`、`marketing_consent`、`constraint`、`policy_note` 会在确认写入后进入 `pending_review`，不会立即进入 customer context。只有审核 approve 后，才会被 `lookup_customer_profile` 返回并参与后续决策。

### Q26：为什么敏感记忆确认后还要 pending_review？

**人话解释**

用户确认只是说“可以记录”，但系统还要防止误识别、医疗风险、隐私合规问题，所以需要人工审核。

**专业回答**

敏感记忆可能影响服务安全或合规，例如过敏、服务禁忌、营销授权。即使用户确认写入，也不应该马上被 Agent 用作自动决策依据。pending_review 机制把“用户允许记录”和“系统允许使用”分开，降低错误识别和敏感信息滥用风险。

### Q27：重复记忆和冲突记忆怎么处理？

**人话解释**

重复的就更新，不要存很多条一样的；冲突的不能自动覆盖，比如用户以前说喜欢安静，现在说喜欢热闹，需要确认。

**专业回答**

重复记忆通过 upsert 更新原记录，提升 evidence、confidence、source_trace 等信息，并增加 version。冲突记忆返回 conflict，不自动覆盖已有内容。这样可以避免长期偏好被模型误判或一次对话随意覆盖，同时保留后续人工确认空间。

### Q28：删除记忆为什么也要走工具确认？

**人话解释**

删除也是敏感操作，误删会影响后续服务体验，所以不能让模型自己删。

**专业回答**

`delete_customer_memory` 属于 sensitive write tool，需要用户确认和 confirmation token 校验。删除采用 soft delete，记录 `deleted_at`、status 和事件，不会物理抹掉审计线索。删除后的记忆不会被 lookup 返回，也不会参与 Agent customer context。

### Q29：LLM 在记忆里做什么？不做什么？

**人话解释**

LLM 可以帮忙识别一些灵活表达，但高风险规则要优先走确定性逻辑，而且 LLM 不能直接写库。

**专业回答**

记忆提取采用规则优先、LLM 补充的策略。过敏、不要营销、删除记忆等高风险或高价值表达由确定性规则识别。LLM 只在规则未命中时补充更灵活的表达识别，并且输出必须被 schema 校验为 `MemoryProposal`。写库仍必须经过用户确认和 Tool Gateway。

## 7. RAG 咨询与知识库

### Q30：RAG 在这个项目里解决什么问题？

**人话解释**

门店政策、价格、迟到取消规则不能靠模型瞎编，要从知识库里查。

**专业回答**

RAG 用于服务咨询和政策问答，例如服务项目说明、价格、取消政策、迟到规则、注意事项等。Agent 判断为 consultation intent 后，会调用知识库检索工具，返回 chunks 和 citation metadata。最终回复需要基于检索结果，并在 trace 中保留来源，方便审计和回放。

### Q31：为什么保留本地 RAG，又支持 MCP RAG？

**人话解释**

本地 RAG 稳定，适合测试；外部 MCP RAG 更接近真实知识服务，适合扩展。

**专业回答**

本地 deterministic knowledge adapter 可以保证 CI、测试和 demo 稳定，不依赖外部服务。MCP RAG 通过环境变量接入外部 RAG server，适合更真实的知识库管理和跨项目复用。两者通过统一的 knowledge tool 暴露给 Agent，避免改动 graph 和 API。

## 8. Trace、Replay 和可观测性

### Q32：为什么要做 trace？

**人话解释**

Agent 出错时不能只看最后一句回复，要知道它每一步怎么判断、调了什么工具、为什么要确认或兜底。

**专业回答**

trace 记录每个节点和工具调用的关键事件，包括 trace_id、conversation_id、node、event_type、timestamp、metadata 和 error。它能帮助定位意图识别、槽位提取、工具规划、确认拦截、RAG 检索和人工升级等问题。对面试来说，trace 证明这个系统不是黑盒聊天，而是可审计的业务 runtime。

### Q33：Replay 有什么用？

**人话解释**

Replay 就是把一次对话的执行过程按顺序复盘出来，方便调试和展示。

**专业回答**

replay 可以从 JSONL trace store 读取同一个 trace_id 的事件，按节点顺序输出摘要。它适合排查工具调用链路、确认写入是否被拦截、RAG 是否命中来源、人工兜底是否被触发，也适合在 demo 或面试中证明系统行为可追踪。

## 9. 流程评测与验证

### Q34：180+ 条测试覆盖了什么？

**人话解释**

不是只测接口能不能通，而是测真实对话场景：缺信息、改时间、技师不可用、敏感记忆、取消政策、人工兜底等。

**专业回答**

评测集覆盖多轮预约、信息缺失、时间修改、技师不可用、档期冲突、重复确认、敏感记忆、记忆删除、迟到/取消政策咨询、RAG grounding、安全拦截和人工兜底等场景。指标包括意图识别、槽位提取、工具选择、工具参数、确认合规、RAG 决策、记忆写入/删除和异常处理。smoke 评测 `failed_case_count=0` 说明当前核心流程在离线场景下通过回归验证。

### Q35：为什么强调 smoke eval，而不说线上提升多少？

**人话解释**

因为没有真实线上数据就不要编。面试里更稳的是讲测试覆盖、场景数、链路验证。

**专业回答**

项目目前更适合用工程验证指标表达，而不是编造业务提升百分比。可以说构建了 180+ 条多轮对话与工具轨迹测试，验证 intent、slot、tool、confirmation、memory、RAG 和 escalation 链路，smoke 评测 failed_case_count=0。这类指标可复现、可解释，比没有数据来源的转化率提升更可信。

## 10. 异常兜底与安全边界

### Q36：什么时候会升级人工？

**人话解释**

只要系统不该自动决定，比如用户说受伤、过敏、投诉退款、工具连续失败、排班不确定，就要交给人工。

**专业回答**

人工兜底触发条件包括医疗或安全风险、投诉退款、低置信度意图、prompt injection、工具参数异常、连续工具失败、排班状态不确定和敏感记忆冲突。触发后系统规划 `escalate_to_human` 工具，并在 trace 中记录 reason、summary 和 requested_action。

### Q37：怎么防止 Agent 产生“假成功”？

**人话解释**

不能工具没成功就说“预约成功”。回复前要检查工具结果。

**专业回答**

系统在响应生成后加入 output policy check。如果回复声称预约已创建，但没有成功的 `create_booking` tool result，就会拦截并改写回复，避免用户看到虚假的业务成功状态。写操作结果必须以工具执行结果为准，而不是以模型生成文本为准。

## 11. 工程取舍类问题

### Q38：这个项目最大的工程难点是什么？

**人话解释**

难点不是某个模型调用，而是让自然语言对话稳定地落到业务流程里，还要能确认、能追踪、能测试。

**专业回答**

最大难点是把不稳定的自然语言输入转化为稳定的业务状态推进。它涉及多轮状态合并、槽位缺失追问、工具调用治理、排班冲突判断、敏感记忆审核、确认 token 和 trace/eval。相比单次问答，这更接近一个带自然语言入口的工作流引擎。

### Q39：如果让你继续优化，你会做什么？

**人话解释**

我会优先做更强的评测、权限和运营后台，而不是先堆更多花哨功能。

**专业回答**

后续我会优先做四件事：第一，扩展 eval dataset，引入更多真实对话边界和对抗样例；第二，为 `/memory` 和写操作管理增加 RBAC 和审批队列；第三，增强排班数据一致性，例如事务、锁或更真实的 schedule repository；第四，完善运营监控，包括工具失败率、人工兜底原因分布、RAG 命中质量和 memory 审核时效。

### Q40：这个项目有什么不足？

**人话解释**

当前更偏工程骨架和离线验证，还不是完整生产系统。

**专业回答**

当前版本已经覆盖核心流程，但还有生产化空间。例如真实门店的排班系统、支付系统、权限系统、用户登录和完整审计报表还没有完全接入；eval 仍以 smoke 回归为主，还需要真实数据分布和人工标注集；memory 审核 UI 是最小管理面，还需要更完整的审批流和权限控制。

## 12. 面试表达模板

### 版本 A：简历项目介绍

面向本地生活门店中咨询、预约、偏好记录和排班校验依赖人工前台的问题，构建业务型 Agent 系统。系统基于单一 `OperationsAgent` 和 LangGraph 状态流，将用户自然语言需求转化为服务咨询、预约状态管理、客户记忆沉淀、排班/技师校验、人工确认和异常兜底等可执行流程，并通过 Tool Gateway、Pydantic schema、confirmation token、trace/replay 和 180+ 条 smoke eval 验证核心链路。

### 版本 B：面试开场说法

我这个项目不是做一个预约 chatbot，而是把门店前台的高频运营流程做成 Agent runtime。用户用自然语言表达咨询、预约、改约、取消或偏好时，系统会先把信息转成结构化状态，再通过 LangGraph 推进流程；真正的查档期、写预约、写记忆都必须走受控工具，并且写操作需要确认 token。最后每一轮都会生成 trace，方便复盘和评测。

### 版本 C：技术亮点说法

技术上我重点做了三层控制：第一是状态控制，用 `booking_slots` 和 `missing_slots` 管理多轮预约；第二是工具控制，用 Tool Gateway 做 schema 校验、权限分级和确认拦截；第三是风险控制，对档期冲突、敏感记忆、工具失败和安全场景做阻断或人工兜底。这样系统不是单纯生成回复，而是能稳定推进业务流程。

## 13. 高频追问速记

### 为什么不是普通 RAG 问答？

因为 RAG 只解决知识问答，不能解决预约状态、多轮补齐、排班校验、写操作确认和客户记忆生命周期。这个项目里 RAG 只是 consultation 工具之一。

### 为什么不是普通工作流？

普通工作流要求输入结构化；这个项目的入口是自然语言，需要 Agent 识别意图、抽取槽位、处理多轮补充和模糊表达，所以是 Agent + workflow 的组合。

### 为什么工具层要 deterministic？

因为业务写入不能依赖模型自由发挥。工具层要用 schema、权限、确认 token 和 trace 约束，保证查档期、写预约、写记忆这些动作可验证。

### 为什么 memory 不直接写？

因为用户偏好可能敏感、过期、冲突或误识别。先生成 proposal，再确认、审核、去重、冲突处理，能降低长期记忆污染风险。

### 为什么 eval 重要？

Agent 项目很容易 demo 看起来能跑，但边界场景不稳定。多轮测试和工具轨迹评测能验证 intent、slot、tool、confirmation、memory、RAG 和 escalation 是否真的串起来。

5. 最后背 Q34-Q40，用评测、兜底和不足证明工程成熟度。

## 15. 深度追问题：面试官二问三问

这一节不是新的项目介绍，而是面试官听完你的回答后，继续往工程细节里追的问题。回答时不要背得太像稿子，重点是把“为什么这么设计”和“边界怎么控制”讲清楚。

### Q41：你说单一 OperationsAgent 更好，那它和传统 Controller + Service 有什么区别？

**人话解释**

Controller + Service 更像固定接口：传什么字段，就按流程处理。OperationsAgent 多了一层“理解和编排”：它要判断用户现在到底想干什么、缺什么信息、该不该调工具、要不要确认。

**专业回答**

传统 Controller + Service 假设请求已经结构化，主要负责参数校验和业务调用。OperationsAgent 面对的是自然语言输入和多轮上下文，核心职责是把不结构化输入转成结构化状态，并决定下一步业务动作。它不是替代 Service，而是在 Controller 和 Tool/Service 之间增加一个状态化编排层：Agent 负责意图判断、槽位推进、工具规划和确认策略；Service 和 Tool 负责确定性执行。

**再追一句**

我不会把数据库写入、排班判断这种确定性逻辑放到 Agent 里。Agent 做编排，工具做执行，这是边界。

### Q42：如果不用 LangGraph，直接写 if else 状态机可以吗？

**人话解释**

可以写，但后面会越来越乱。LangGraph 的价值是让流程节点清楚，方便测试和 trace。

**专业回答**

如果流程很短，用 if else 可以。但这个项目包含意图识别、记忆加载、槽位提取、工具规划、确认拦截、工具执行、回复生成和输出检查，节点之间有明确状态流转。用 LangGraph 可以把这些步骤显式建模，每个节点只负责一类状态变更，trace 也能按节点记录。相比手写 if else，LangGraph 更利于维护、测试和扩展。

**反问式补充**

我选 LangGraph 不是因为它更“智能”，而是因为它更适合把 Agent 流程工程化。

### Q43：OperationsAgent 会不会变成上帝类？

**人话解释**

风险存在，所以要靠节点拆分和工具下沉来控制。Agent 不是什么都自己做。

**专业回答**

这是单一编排层需要注意的风险。我的处理方式是：OperationsAgent 只持有统一状态和流程编排，不直接实现所有业务能力；具体能力拆成 graph nodes 和 tools。比如排班校验、知识库检索、客户记忆写入都通过工具边界完成。这样 Agent 是 orchestration layer，不是 God object。

**面试加分点**

如果后续某个节点变复杂，比如排班优化或记忆审核，可以继续拆成独立 service 或 subgraph，而不是让主 Agent 膨胀。

### Q44：多轮对话里怎么避免旧状态污染新请求？

**人话解释**

不能一直沿用旧草稿。要看用户是不是还在同一个预约任务里，还是开启了新任务。

**专业回答**

状态污染主要来自跨轮复用 `booking_slots`。工程上需要结合 intent、conversation_id、用户当前表达和确认状态判断是否复用旧槽位。比如用户继续补充时间、技师、时长，就增量合并；如果用户开启新的预约或取消当前操作，就应该清空或切换上下文。同时字段来源 `booking_slot_sources` 可以帮助判断某个字段来自上一轮、当前用户输入还是 memory，避免无意识继承旧值。

**可承认不足**

当前版本已经支持多轮补充和字段来源，但更复杂的多任务并行对话还可以继续增强，例如引入 task_id 或 pending intent stack。

### Q45：如果用户一句话里同时有咨询和预约，怎么处理？

**人话解释**

要看哪个是主任务。如果用户问完马上要约，就预约优先；如果只是问规则，就走知识库。

**专业回答**

混合意图需要根据业务优先级处理。预约、改约、取消这类会改变状态的任务优先级更高，因为它们需要槽位管理和确认链路；政策、价格、注意事项走 RAG 咨询路径。工程上可以先识别主 intent，再在回复里补充必要知识信息。对于强混合场景，也可以拆成先回答咨询，再继续推进预约的多步流程。

**一句话背诵**

我的原则是：会影响业务状态的意图优先进入状态机，纯知识问题进入 RAG。

### Q46：槽位抽取如果错了怎么办？

**人话解释**

不能直接写。错了也应该在确认摘要里暴露出来，让用户确认前发现。

**专业回答**

槽位抽取错误通过多层机制降低风险。第一，关键字段不完整时不会进入写工具；第二，进入写工具前会生成 confirmation summary，让用户确认服务、日期、时间、技师和特殊要求；第三，写操作需要 confirmation token；第四，写入前还会进行档期校验。如果用户发现摘要不对，可以在下一轮修改字段，而不是直接落库。

**面试追问点**

如果要进一步优化，可以给每个槽位加 confidence 和 source，低置信度字段强制追问。

### Q47：为什么不让 LLM 直接输出最终工具调用 JSON？

**人话解释**

LLM 可以提建议，但不能当最终执行凭证。最终要过 schema、权限、确认。

**专业回答**

LLM 直接输出工具 JSON 风险比较高：可能缺字段、字段类型错、绕过确认、或者把用户没确认的信息当成事实。我的设计是 LLM/规则层可以参与意图识别和 proposal 生成，但工具调用必须经过 Tool Gateway。Gateway 用 Pydantic schema、permission、requires_confirmation 和 token 校验做最后约束。

**一句话背诵**

模型负责“建议下一步”，Gateway 决定“能不能执行”。

### Q48：confirmation token 真的有必要吗？用户说“确认”不就行了吗？

**人话解释**

只说确认不够，因为要确认“哪一个操作、哪些参数”。否则容易串单。

**专业回答**

confirmation token 的价值是绑定具体 conversation、tool_name 和 arguments。没有 token 的话，用户一句“确认”可能被错误地绑定到旧的 pending action，或者参数在确认前后被模型改写。token 让确认成为一个可验证的业务动作，而不是一个自然语言猜测。

**工程细节说法**

确认不是确认一句话，而是确认一个结构化 action。

### Q49：如果用户确认后又反悔怎么办？

**人话解释**

确认前可以直接取消；确认后如果已经写入，就要走取消或改约流程，不能偷偷回滚。

**专业回答**

如果写操作还没有执行，用户拒绝确认时系统会清空 confirmation_request，不调用写工具。如果写操作已经执行，比如预约已创建，就需要走业务上的取消或改约工具，而不是直接撤销数据库状态。这样可以保留审计链路，也符合真实业务流程。

### Q50：档期冲突为什么不能只靠数据库唯一约束？

**人话解释**

因为时间段不是简单字段相等。两个预约开始时间不同，也可能重叠。

**专业回答**

数据库唯一约束适合处理完全相同的 key，比如同一技师同一开始时间。但预约冲突是时间区间重叠问题，需要判断 `new_start < existing_end AND new_end > existing_start`。这类约束通常需要业务层校验、数据库事务或更复杂的 exclusion constraint。当前项目在写入前进行二次 schedule check，用业务规则阻断重叠时间段。

**继续追问**

如果生产化，我会进一步加入事务、锁或数据库层约束，防止并发写入绕过应用层检查。

### Q51：如果两个用户同时抢同一个技师时间，你现在的方案够吗？

**人话解释**

现在适合 demo 和离线验证。生产环境还要加事务和锁。

**专业回答**

当前方案强调写入前二次校验，能覆盖常规流程，但生产级并发还需要更强一致性控制。比如在数据库层对技师和时间区间做事务保护，写入时加锁，或使用专门的 schedule repository 保证 check-and-write 原子性。否则两个请求可能同时通过读校验，再同时写入。

**坦诚回答**

面试里可以说：当前版本把冲突判断和阻断链路建出来了，但生产化还需要数据库事务和并发控制增强。

### Q52：客户记忆为什么不用简单 key-value 存偏好？

**人话解释**

因为记忆不是简单配置。它有证据、来源、敏感性、是否过期、是否审核、版本和删除记录。

**专业回答**

简单 key-value 只能存最终值，无法表达记忆生命周期。这个项目里的 memory 需要记录 evidence、confidence、sensitivity、source_trace_id、source_conversation_id、expires_at、review_status、version 和 deleted_at。这样才能处理敏感审核、冲突、重复更新、软删除和审计回放。

### Q53：怎么避免客户记忆越积越多、越用越错？

**人话解释**

要有过期、审核、删除和冲突处理，不能无限相信旧记忆。

**专业回答**

主要通过四类机制控制：第一，expires_at 让过期记忆不进入 lookup；第二，敏感记忆 pending_review，未审核不使用；第三，重复记忆 upsert 更新而不是新增；第四，删除是 soft delete，用户可以撤销或审计。后续还可以加入记忆权重衰减、最近使用时间和人工定期清理。

### Q54：如果记忆和用户本轮输入冲突，听谁的？

**人话解释**

听用户本轮输入。本轮明确表达优先级最高。

**专业回答**

实时用户输入优先于历史记忆。memory 是辅助上下文，不应该覆盖用户当前明确意图。例如历史记忆是喜欢安静房间，但用户本轮说“这次想热闹一点”，应以本轮输入为准，并可以触发冲突记忆处理或临时 special_requests 更新。字段来源记录也能帮助确认这个优先级。

**一句话背诵**

记忆是默认偏好，不是强制规则；用户本轮明确输入优先。

### Q55：敏感记忆 pending_review 会不会影响体验？

**人话解释**

会稍微慢一点，但换来安全。比如过敏这种信息，宁可人工确认后再自动使用。

**专业回答**

pending_review 是体验和风险之间的取舍。普通偏好可以确认后直接用于后续建议；但过敏、服务禁忌、营销授权这类信息涉及安全和合规，如果错误使用风险更高。因此我把“记录下来”和“允许 Agent 使用”拆成两个阶段。这样短期增加审核成本，但能降低错误个性化和敏感信息误用。

### Q56：RAG 检索结果不准怎么办？

**人话解释**

不能硬答。要么告诉用户无法确认，要么升级人工或提示以门店规则为准。

**专业回答**

RAG 不准主要有两类：没有召回相关内容，或召回内容不属于目标知识域。系统可以通过 citation metadata 检查 source、chunk_id 和 score。若没有可靠来源，回复应避免编造政策，可以转人工或给出保守说明。评测中也要区分 rag_used 和 groundedness，不能只看是否调用了检索工具。

### Q57：trace 记录会不会太多？怎么控制价值？

**人话解释**

trace 不是什么都记，而是记关键决策点：意图、槽位、工具、确认、错误。

**专业回答**

trace 的目标是复盘业务决策，不是保存所有中间文本。关键事件包括节点名称、事件类型、工具名、状态、错误码、memory_id、rag source、confirmation_required 等。这样既能支持调试和审计，又避免把 raw prompt 或敏感内部异常暴露出去。

### Q58：eval 里 failed_case_count=0 能说明什么？不能说明什么？

**人话解释**

说明当前测试集都过了，不代表线上一定没问题。

**专业回答**

`failed_case_count=0` 说明在当前 180+ 条 smoke cases 里，核心链路通过了离线回归，包括 intent、slot、tool、confirmation、memory、RAG 和 escalation。但它不能代表真实线上分布，也不能证明模型在所有表达上都正确。它的价值是提供可复现的工程质量基线，而不是替代线上监控和真实用户反馈。

### Q59：你怎么证明这个系统比 prompt-only 更可靠？

**人话解释**

因为它不是靠一段 prompt 记住所有规则，而是把规则拆到状态、工具、确认和测试里。

**专业回答**

prompt-only 方案很难稳定保证多轮状态、工具权限、写入确认和异常兜底。这个项目把可靠性拆到工程层：状态由 booking_slots 显式维护，写操作由 Tool Gateway 拦截，敏感记忆有审核状态，排班冲突由工具校验，trace 记录执行过程，eval 覆盖边界场景。这些机制降低了模型输出波动带来的业务风险。

### Q60：如果面试官问“你这个项目最有含金量的点是什么”，怎么答？

**人话解释**

不要说“用了 LangGraph”这么浅。要说你把一个真实业务闭环做成了受控 Agent 流程。

**专业回答**

我认为最有含金量的是把自然语言入口、业务状态机、受控工具调用、确认写入、客户记忆、排班冲突和 trace/eval 串成了完整闭环。它不是单点模型调用，而是一个能把用户需求推进到真实业务动作的 Agent runtime，并且对写操作、安全边界和评测验证做了工程约束。

## 16. 面试官追问总表（去重强化版）

> 使用方式：这一节把新增追问里和前文重复的内容做了合并。背的时候不要逐字背，按“业务问题 -> Agent 边界 -> 状态机 -> 工具治理 -> 预约一致性 -> 记忆安全 -> 评测验证”这条线讲。

### Q61：这个 Agent 最核心解决的业务问题到底是什么？

**人话解释**

不是“做一个能聊天的客服”，而是解决门店前台工作里最容易出错的几件事：用户说得零散、前台反复追问、偏好记不住、档期查错、写入前没确认、出问题后难复盘。

**专业回答**

这个系统解决的是本地生活门店的 booking operations 问题：把用户自然语言里的咨询、预约、改约、取消、偏好记录和异常反馈，转成可控的业务流程。核心价值不是提升模型聊天能力，而是把多轮沟通中的状态、工具调用、排班校验、客户记忆、人工确认和 trace/eval 串起来，减少信息散落、重复追问、误写库和不可追溯的问题。

**面试补充**

如果被问“上线前后有没有量化指标”，要诚实回答：当前更适合说工程验证指标，例如 180+ 多轮测试、failed_case_count=0、slot/tool/confirmation/memory/RAG 覆盖。咨询转预约率、平均响应时间、人工介入率、冲突率这些是上线后应监控的业务指标，不能没有数据源就编数字。

### Q62：为什么这个问题适合用 Agent，而不是普通表单或规则系统？

**人话解释**

表单适合用户知道自己要填什么。这个场景里用户经常说“明天下午做上次那个”“找个轻一点的师傅”“不要太晚”，信息不完整、表达模糊，还会中途改主意，所以需要 Agent 做理解和状态推进。

**专业回答**

普通规则系统适合固定入口和明确字段，但门店预约是多轮、上下文相关、带不确定性的任务。Agent 的价值在于理解自然语言、识别意图、抽取和合并槽位、根据缺失字段追问、调用工具查档期或知识库，并在写操作前进入确认流程。系统仍然保留确定性后端规则：工具权限、schema 校验、排班二次校验、confirmation token 和 trace 都不依赖 LLM 自觉。

### Q63：这个系统的用户是谁？不同角色怎么区分？

**人话解释**

主要入口面向消费者，但它不是只给消费者看的聊天框。真正被优化的是门店运营流程，前台、店长和人工客服都可能使用后台信息。

**专业回答**

当前实现以 `/api/operations/chat` 这类 operations chat 入口为主，模拟消费者通过网页客服或未来微信、小程序、电话转写等渠道进入。消费者发起咨询和预约，Agent 负责收集信息和生成确认请求。门店员工侧更适合承担审核、人工接管、敏感记忆 approve/reject、异常处理和排班配置维护。生产化时应进一步拆 RBAC：前台可处理预约和普通偏好，店长可处理敏感审核和高风险异常，技师侧只暴露与自己相关的排班和服务信息。

### Q64：Agent 的职责边界是什么？

**人话解释**

Agent 可以帮用户推进流程，但不能“想写就写”。只要涉及创建预约、改约、取消、删除记忆、写敏感偏好，都必须走确认或审核。

**专业回答**

Agent 的职责是理解请求、维护状态、计划工具调用、生成确认话术和解释工具结果。它不能绕过 Tool Gateway 直接写库，不能伪造 confirmation token，不能执行 SQL，也不能把用户一句“确认”当作万能授权。创建、改约、取消、写客户记忆、删除记忆属于写操作，必须经过后端生成的确认上下文。涉及医疗风险、退款投诉、强情绪、排班不确定、敏感信息、用户要求人工时，应升级人工。

### Q65：为什么设计成单一 OperationsAgent，而不是多个 Agent？

**人话解释**

因为真实前台对话不是“咨询归咨询、预约归预约、记忆归记忆”那么干净。用户经常一句话同时问政策、约时间、表达偏好。单一 OperationsAgent 更容易维护同一份状态。

**专业回答**

当前选择单一 OperationsAgent，是因为核心难点是同一轮对话里的状态一致性，而不是多个专家 Agent 的协作。咨询、预约、记忆、确认和兜底都围绕同一个 conversation state 推进。单 Agent 的好处是上下文统一、状态合并简单、trace 连续；坏处是状态字段会膨胀，节点边界需要更清楚。控制方式是把编排留在 Agent，把具体能力下沉到 Tool Gateway、schema、store 和 policy 节点，避免把所有业务逻辑塞进一个 prompt。

### Q66：LangGraph 的节点可以怎么讲？

**人话解释**

可以把它讲成一条门店前台工作流：先接收用户话，再判断要做什么，再补信息，再决定查什么工具，再决定能不能写，最后生成回复和记录 trace。

**专业回答**

可以按当前工程口径这样讲：`initialize_turn` 初始化会话和 trace；`classify_intent` 识别咨询、预约、改约、取消、偏好、人工等意图；`load_customer_context` 读取可用客户记忆；`extract_booking_slots` 抽取并合并 booking slots；`propose_memory_writes` 生成 MemoryProposal；`plan_tool_calls` 决定是否查知识库、查档期、查客户信息或准备写操作；`execute_tools` 统一经过 Tool Gateway；`generate_response` 生成追问、确认或结果回复；`output_policy_check` 做最终安全和暴露控制；`finalize_turn` 写 trace。概念上也对应 `ask_missing_info`、`check_availability`、`request_confirmation`、`handoff_to_human` 和 `write_trace`。

### Q67：LangGraph state 里保存什么？为什么不能只靠聊天历史？

**人话解释**

聊天历史是流水账，state 是办事清单。用户说过很多话，但系统真正要知道的是项目、日期、时间、技师、偏好、缺什么、查过什么、能不能确认。

**专业回答**

OperationsAgent state 至少需要保存 `user_id`、`conversation_id`、`message`、`intent`、`booking_slots`、`missing_slots`、`tool_plan`、`tool_results`、`confirmation_required`、`confirmation_request`、`reply`、`trace_id`、`trace_events`、`errors`，以及客户记忆和 RAG 使用标记。显式 state 的价值是让每个节点读写稳定字段，而不是让 LLM 每轮从聊天记录里重新猜。这样也方便测试、回放和定位是哪一步把状态弄错了。

### Q68：多轮信息怎么合并？比如“周五做美甲”“下午吧”“还是小王”“不要太晚”。

**人话解释**

每一轮不是重新开始，而是往同一张预约草稿里补字段。第一轮补项目和日期，第二轮补时间段，第三轮补技师，第四轮补特殊偏好。

**专业回答**

系统会把每轮用户输入解析为 partial slots，然后和已有 `booking_slots` 做增量合并。明确新值覆盖旧值，模糊表达先进入待确认或 special request，不直接写成确定时间。字段来源需要记录，例如 `date` 来自第一轮，“time_window=afternoon”来自第二轮，“preferred_staff=小王”来自第三轮，“not too late”来自第四轮。当前项目更强调 field source，字段级 confidence 可以作为后续增强，用来决定低置信槽位是否必须追问。

### Q69：旧槽位和新槽位冲突时怎么处理？

**人话解释**

用户当前明确说的话优先，但系统不能偷偷改完就写库。关键字段变了，就要重新查档期、重新生成确认。

**专业回答**

如果用户把“明天下午 3 点”改成“还是 4 点吧”，应保留服务项目、用户身份、技师等未冲突字段，但时间字段更新，旧 confirmation token 失效，并重新查档期。如果用户说“那不要小李了”，不能直接判断为取消预约，需要根据上下文区分：不指定技师、换技师、对小李负向偏好，还是取消整个预约。策略是优先结合当前 intent、最近 confirmation draft 和字段来源判断；不确定时追问，而不是自动写入负向记忆或取消预约。

### Q70：booking_slots 的结构应该怎么讲？

**人话解释**

它就是预约草稿，不是最终订单。里面放用户已经说清楚的内容，也放还没补齐的内容。

**专业回答**

可以按概念结构讲：`service_type` 或 `service_item` 表示服务项目，`date` 表示预约日期，`time_window` 或 `start_time/end_time` 表示时间段，`duration_minutes` 表示时长，`preferred_staff` 表示指定技师，`special_requests` 表示安静、手法轻、不要太晚等偏好，`customer_name` 和 `user_id` 表示身份，改约还需要 `booking_id`、`new_date`、`new_time_window`。生产级还应记录 `field_sources`、`confidence`、`normalized_datetime` 和 `catalog_match`。服务时长一般以服务目录配置为准，用户说的时长只能作为偏好或需要确认的覆盖请求。

### Q71：什么叫关键字段完整？

**人话解释**

能不能写预约，不看用户说得热不热情，而看必须字段有没有齐。缺项目、缺日期、缺时间，就只能追问。

**专业回答**

创建预约通常需要用户身份、门店上下文、服务项目、日期、时间段、服务时长和用户确认；技师可以是指定技师，也可以由系统推荐。改约需要已有 `booking_id`、目标日期或时间、用户确认和重新排班校验。取消预约需要 `booking_id`、用户身份和确认。偏好记录需要客户身份、MemoryProposal、证据文本、类型和确认或审核状态。咨询类请求不一定进入 booking_slots，但如果咨询里包含预约意图，就要同时进入槽位流程。

### Q72：工具清单怎么回答最清楚？

**人话解释**

按“查信息”和“改数据”分。查信息可以直接做，改数据必须确认。

**专业回答**

可以这样背：

| Tool | 作用 | 是否写库 | 是否需要确认 |
| --- | --- | --- | --- |
| `search_service_kb` / `search_knowledge_base` | 服务、政策、注意事项咨询 | 否 | 否 |
| `query_technician` / `find_available_staff` | 查技师和可用人员 | 否 | 否 |
| `check_availability` / `check_schedule` | 查档期、忙闲和冲突 | 否 | 否 |
| `recommend_alternative` | 推荐替代技师或时间 | 否 | 否 |
| `create_booking` | 创建预约 | 是 | 是 |
| `reschedule_booking` | 改约 | 是 | 是 |
| `cancel_booking` | 取消预约 | 是 | 是 |
| `write_customer_preference` | 写客户偏好 | 是 | 视类型确认或审核 |
| `delete_customer_memory` | 删除记忆 | 是 | 是 |
| `handoff_to_human` | 人工兜底 | 可写工单 | 通常不需要用户二次确认 |

重点不是工具多，而是所有工具都经过统一 Gateway、schema、permission、confirmation 和 trace。

### Q73：Pydantic schema 能防住什么，防不住什么？

**人话解释**

schema 能防止参数乱填，但不能证明业务一定正确。

**专业回答**

Pydantic schema 可以限制字段类型、必填字段、枚举值、时间格式、参数结构和部分业务约束。例如 `create_booking` 不能缺少服务、日期、时间和用户；`booking_id` 不能是任意对象；permission 和 confirmation 字段必须符合工具定义。它防不住语义真实性，例如“这个技师是否真的可用”“用户是不是确实想取消”“知识库答案是否最新”，也防不住并发竞争。所以 schema 是第一层约束，后面还要有 catalog 校验、排班工具、确认 token、事务和 trace。

### Q74：权限分级怎么讲？

**人话解释**

不是所有工具都一个级别。查一下政策和取消用户预约，风险完全不同。

**专业回答**

可以分四层：Level 0 是 read-only 查询工具，例如知识库、技师、档期、客户上下文 lookup；Level 1 是推荐类工具，例如替代时间和技师排序；Level 2 是普通写入工具，例如创建预约、改约、取消、普通偏好写入，需要用户确认和 token；Level 3 是高风险或敏感写入，例如服务禁忌、营销授权、删除记忆、投诉退款相关动作，需要更强确认或人工审核。LLM 只能提出计划，不能绕过 Gateway 直接写库。

### Q75：confirmation token 该怎么设计？

**人话解释**

token 是后端给的“这次确认凭证”，不是 LLM 编出来的一句话。用户改了时间或技师，旧凭证就不能用了。

**专业回答**

confirmation token 应由后端生成，绑定 `user_id`、`conversation_id`、操作类型、工具名、关键参数 hash、draft_id、过期时间和一次性使用状态。LLM 只展示确认内容，不能自造 token。用户修改关键字段后，旧 token 必须失效并重新生成。生产级还应加入 idempotency key，防止网络重试或重复点击导致重复写入。token 校验必须发生在 Tool Gateway 或后端服务层，不能只靠 prompt 判断“用户说了确认”。

### Q76：如何避免 LLM 绕过规则直接写库？

**人话解释**

不要把安全寄托在“模型会听话”。模型就算输出 SQL 或 JSON，也不会被系统直接执行。

**专业回答**

系统通过四层控制避免绕过：第一，工具 allowlist，只有注册工具能被调用；第二，所有调用走 Tool Gateway，统一做 schema、permission、confirmation 校验；第三，写库工具只接受服务端生成的 draft 或 confirmation token；第四，trace 记录工具名、参数摘要、结果和错误。LLM 输出的自然语言、SQL 或任意 JSON 都不是数据库指令。业务安全在后端边界实现，而不是依赖 prompt injection 防护词。

### Q77：档期冲突算法怎么讲？

**人话解释**

不是开始时间一样才冲突，只要两个时间段重叠就冲突。

**专业回答**

时间区间重叠判断可以用：`new_start < existing_end AND new_end > existing_start`。已有预约 10:00-11:00，新预约 10:30-11:30 冲突；新预约 11:00-12:00 通常不冲突，因为刚好接上。生产规则还要考虑技师休息时间、门店闭店时间、服务前后 buffer、跨天营业时间、多资源占用，比如房间加技师，以及连续多服务组合。当前项目重点是把冲突判断、阻断写入和替代建议链路跑通。

### Q78：为什么写库前还要二次查档期？

**人话解释**

用户确认前查到有空，不代表用户确认那一秒还一定有空。别人可能先约走了。

**专业回答**

第一次查档期用于生成可选方案，最终写入前必须二次校验，防止确认窗口期间排班变化。生产级需要把二次校验和写入放进同一个一致性边界里，例如数据库事务、行锁、乐观锁、排他约束或 schedule repository 的原子 check-and-write。否则两个用户可能同时通过读校验，再同时写入。当前项目实现了写前阻断思路，生产化重点是补强并发控制。

### Q79：指定技师不可用时，替代推荐怎么做？

**人话解释**

不要直接说“约不了”。应该给用户可选择的替代方案。

**专业回答**

推荐策略可以按优先级排序：同一时间其他可用技师、同一技师临近时间、同服务等级技师、用户历史偏好技师、门店推荐技师。推荐数量不宜太多，通常 2 到 3 个最清楚。替代方案不能直接写入，仍然需要用户确认。如果用户坚持原技师原时间，系统应该说明当前不可用，并提供排队、稍后提醒或人工处理选项。

### Q80：什么叫排班结果“不确定”？

**人话解释**

不是只有“有空”和“没空”。如果系统查不到、数据不完整、服务时长不确定，也不能贸然写入。

**专业回答**

不确定场景包括排班工具超时、技师信息缺失、busy schedule 数据延迟、服务时长无法确定、门店营业时间跨天异常、多个资源只校验了一部分、知识库或配置版本不一致。处理原则是阻断写入，不伪造成功；用户侧表达可以是“我现在还不能确认这个时段可约，我可以为你查临近时间或转人工确认”，而不是暴露内部异常堆栈。

### Q81：MemoryProposal 是什么？为什么不能直接写 memory？

**人话解释**

MemoryProposal 是“建议记录一条记忆”，不是“已经写入并生效”。中间还要确认、审核、去重和冲突判断。

**专业回答**

MemoryProposal 至少应包含 `customer_id`、`memory_type`、`content`、`evidence_text`、`source_conversation_id`、`source_trace_id`、`confidence`、`action`、`review_required`、`expires_at`。LLM 可以负责识别“这句话可能是偏好或禁忌”，但输出必须落到结构化 proposal。真正写库仍然走 Tool Gateway。普通偏好可在用户确认后写入；敏感记忆写入后为 `pending_review`，approve 前不进入 customer context。

### Q82：普通偏好和敏感记忆怎么区分？

**人话解释**

“喜欢安静”一般是普通偏好；“皮肤过敏”“不要营销”“删除我的记忆”风险更高，要更谨慎。

**专业回答**

普通偏好包括安静环境、手法轻一点、喜欢某类项目、偏好某位技师等，通常用于推荐排序。敏感或高风险记忆包括过敏、怀孕、服务禁忌、医疗风险、营销授权或拒绝、删除请求、强烈负向偏好和隐私限制。比如“不要男技师”可以作为服务偏好，但涉及敏感属性时要谨慎表达和使用；“不要再发营销短信”必须作为高优先级营销拒绝记录；“我不想让店员知道这个偏好”说明不能默认暴露给人工界面。

### Q83：冲突记忆怎么处理？

**人话解释**

不能把旧记忆直接覆盖掉。因为“以前喜欢小李，现在不要小李”可能代表偏好变化，也可能只是这一次不想约。

**专业回答**

冲突记忆要区分本轮临时偏好和长期偏好更新。历史记忆是“喜欢小李”，新对话说“以后不要给我约小李了”，更像长期负向偏好，应生成更新或冲突 proposal，并经确认后写入。历史是“不接受营销”，这次说“有活动可以告诉我”，营销授权涉及合规，必须明确确认来源、时间和渠道，不能自动覆盖。推荐时本轮明确输入优先，已审核的负向偏好和服务禁忌是强约束，普通偏好是 soft ranking。

### Q84：记忆怎么用于后续预约？

**人话解释**

记忆不是拿来替用户做决定，而是减少重复沟通。比如用户偏好安静，就默认放到备注或推荐里；但如果用户本轮说不要，就听本轮。

**专业回答**

记忆主要在三个阶段使用：槽位补全阶段，用历史偏好解释“上次那个项目”；推荐阶段，用偏好影响技师或时间排序；安全阶段，用服务禁忌和营销拒绝作为强约束。系统不应该把所有记忆原样暴露给用户或技师，只应转化为必要的业务上下文。expired memory 不进入 lookup，pending_review 不进入 customer context，soft deleted memory 只保留审计，不参与推荐。

### Q85：什么时候转人工？人工接管时带什么？

**人话解释**

转人工不是失败，而是风险控制。机器不确定、用户生气、涉及退款投诉或敏感风险时，让人接。

**专业回答**

转人工规则包括多轮无法理解、工具连续失败、排班结果不确定、用户强情绪、退款赔偿投诉、医疗或服务安全风险、敏感记忆审核、用户主动要求人工、金额或服务风险较高。交给人工时应带结构化摘要：用户原话、当前 intent、booking_slots、缺失字段、工具调用结果、失败原因、confirmation 状态、memory proposal、trace_id 和建议下一步。人工处理结果也应回写 trace，避免 Agent 后续不知道已经处理过。

### Q86：重复确认怎么防止重复创建预约？

**人话解释**

用户点两次确认或发两次“确认”，不能创建两单。

**专业回答**

确认流程要有一次性 token 和幂等控制。`create_booking` 成功后，confirmation token 应标记 consumed，重复使用直接返回“已处理”或查询已有预约结果，而不是再次写库。生产级还应使用 idempotency key，例如基于 `user_id + operation_type + draft_id + token_hash` 生成，保证网络重试、前端重复点击、Agent 重放都不会重复创建。trace 中应记录 duplicate confirmation 或 idempotent replay。

### Q87：服务咨询和预约混在一句话里怎么办？

**人话解释**

先回答必要咨询，再继续预约。但如果咨询涉及安全风险，就不能直接进入预约。

**专业回答**

例如用户说“这个护理适合敏感肌吗？如果可以帮我约明天下午”，系统应识别为多意图：咨询加预约。RAG 先检索服务说明和注意事项，如果知识库明确说明适合条件或需要试敏，则回复 grounded answer，再收集预约槽位。如果涉及过敏、怀孕、医疗风险或知识库缺失，应转人工或建议门店确认，而不是直接推进写入。多意图处理的关键是不要让预约流程覆盖安全咨询。

### Q88：迟到、取消、退款政策怎么回答？

**人话解释**

政策问题不能靠模型编。查得到就按来源答，查不到就转人工或说需要门店确认。

**专业回答**

迟到、取消、退款、投诉类问题应该来自门店知识库或政策配置，最好带门店、版本和更新时间。不同门店政策可能不同，RAG 查询时要带 store context。如果没有可靠来源，Agent 不应该编赔偿、退款或免责规则。用户问退款、赔偿、投诉升级时，应该转人工或生成工单。回复里可以保守说明“我需要为你确认门店当前政策”，而不是输出无来源承诺。

### Q89：180+ 条测试 case 的 golden answer 应该是什么形式？

**人话解释**

不是只看程序有没有报错，而是看业务动作有没有做对。

**专业回答**

golden case 应包含多轮用户输入、期望 intent、期望 booking_slots、期望 missing_slots、期望工具调用和参数摘要、是否需要 confirmation、是否禁止写库、是否使用 RAG、是否生成 MemoryProposal、是否升级人工、期望回复要点。负样本也很重要，例如 prompt injection、信息缺失却要求创建、未确认写操作、敏感记忆直接使用、排班冲突仍写入。`failed_case_count=0` 只能说明当前测试集通过，不代表线上长尾、并发和真实渠道都稳定。

### Q90：意图识别、槽位抽取和工具选择分别怎么评估？

**人话解释**

要分开评估。意图错、字段错、工具错是三种不同问题，不能只看最终回复像不像。

**专业回答**

意图识别可以用标签级 precision/recall 或 case pass rate，重点看预约、改约、取消、咨询、偏好、人工兜底、多意图混合。槽位抽取做字段级准确率，例如日期时间、服务项目、技师、时长、备注、booking_id，并单独评估“明天下午”“这周五”“上次那个”“不要太晚”等模糊表达。工具选择看 should-call 和 should-not-call：什么时候查知识库、什么时候查档期、什么时候写记忆、什么时候禁止 `create_booking`。工具参数错误、漏确认、错用写工具都应算失败。

### Q91：整体技术架构怎么画？

**人话解释**

从用户入口到 Agent，再到工具网关，再到业务系统，最后到 trace 和 eval。

**专业回答**

可以这样讲：

```text
用户渠道（Web / 微信 / 小程序 / 电话转写）
  -> 会话服务
  -> OperationsAgent / LangGraph
  -> Tool Gateway
  -> 预约系统 / 技师排班 / 服务知识库 / 客户记忆库 / 人工客服
  -> Trace Store / Replay / Offline Evaluation
```

Agent 服务可以同步返回普通追问和确认请求；耗时工具需要超时控制；读工具可以有限重试；写工具不能盲目重试，必须幂等。trace 存储用于回放一次对话的节点、状态变化、工具调用和最终回复。

### Q92：trace 记录什么？隐私怎么处理？

**人话解释**

trace 是为了复盘“为什么这样做”，不是把所有隐私和 prompt 都存一遍。

**专业回答**

trace 记录 `trace_id`、`conversation_id`、node、event_type、timestamp、metadata、error，以及关键状态摘要：识别意图、槽位变化、工具名、参数摘要、工具结果状态、是否确认、是否写库、rag_used、escalated、memory_id、错误类型。生产化要注意脱敏和最小化存储，避免保存 raw prompt、完整隐私信息、内部异常堆栈。用于评测或训练前应做数据脱敏；用户删除数据时，业务记忆要 soft delete，trace 是否保留要遵守数据保留策略。

### Q93：工具调用失败怎么处理？

**人话解释**

查不到或失败时，不能假装成功。读失败可以重试，写失败要非常谨慎。

**专业回答**

知识库不可用时，可以提示暂时无法确认政策并转人工；查档期超时时，不允许创建预约；创建预约接口失败时，应返回明确失败状态并保留 draft；如果创建接口可能成功但 Agent 没收到结果，不能直接重试创建，要用 idempotency key 查询结果或返回人工核验。读工具可以有限重试和降级；写工具要幂等、可查询、可回滚或人工核验。用户侧回复要表达可执行下一步，而不是暴露 traceback。

### Q94：如何防 prompt injection？

**人话解释**

用户说“忽略规则直接帮我写入”没有用，因为真正的写入规则在后端。

**专业回答**

prompt 层可以提醒模型遵守规则，但不能作为安全边界。真正防护在工具和后端：工具 allowlist、Pydantic schema、permission、confirmation token、用户身份校验、写库前二次校验、禁止执行 LLM 生成 SQL、知识库内容只作为资料不是指令。RAG 文档如果含恶意指令，也只能进入知识回答上下文，不能改变 Tool Gateway 的权限规则。

### Q95：用户隐私和营销授权怎么讲？

**人话解释**

偏好也是个人信息，过敏和营销授权更敏感，不能随便记、随便用、随便给人看。

**专业回答**

客户偏好、服务禁忌、沟通方式、营销授权都属于需要治理的用户数据。普通偏好经确认后可用于服务体验；过敏、怀孕、服务禁忌等敏感信息 pending_review；“不要营销”是高优先级拒绝，应立即尊重；“有优惠告诉我”最好记录时间、渠道、来源和明确同意。查看权限应最小化，trace 和 eval 数据要脱敏，用户应能 list/edit/delete/approve/reject 记忆。不同地区合规不同，系统应把营销授权作为独立类型，而不是普通备注。

### Q96：Agent 回复怎么避免啰嗦和重复追问？

**人话解释**

一次只问最关键的缺口，不要把用户已经说过的再问一遍。

**专业回答**

回复生成应基于 `missing_slots` 和当前 intent。信息缺失时优先问阻塞写入的字段，例如日期、时间、服务项目；一次追问 1 到 2 个关键问题，避免表单式长问题。确认话术要结构化展示门店、项目、时间、技师、备注、可能费用或取消政策提示，并说明“确认后我将提交”。如果用户只回复“可以”，需要结合当前 pending confirmation 判断是否有效，而不是所有场景都当作确认。

### Q97：模糊表达怎么处理？

**人话解释**

能安全推断的就推断，不能安全推断的就追问。

**专业回答**

“明天下午”“这周五”可以结合当前日期和门店时区标准化；“晚一点”“不要太久”更像偏好或时间范围，需要追问具体时间；“上次那个”需要查历史预约或客户记忆；“手法轻一点”可以作为 special request 或普通偏好 proposal；“别给我安排上次那个”需要区分临时不指定、负向偏好和投诉；“你看着安排吧”可以触发推荐，但写入前仍要展示具体方案并确认。原则是：低风险偏好可辅助推荐，高风险或关键字段必须确认。

### Q98：这个项目最难的部分是什么？

**人话解释**

难点不是接个模型，而是让模型参与业务流程时不乱写、不漏信息、出了错能查。

**专业回答**

最难的是多轮状态合并、工具调用边界、确认写入一致性和客户记忆安全分类。多轮对话里用户会补充、反悔、改时间、换技师，状态必须能增量更新且不污染。写库动作必须被 Gateway、schema、confirmation 和排班校验约束。记忆既要提高体验，又要处理敏感、过期、冲突、删除和审核。最后还要用 trace 和 eval 把这些链路变成可验证工程，而不是靠演示效果。

### Q99：你做过哪些 trade-off？

**人话解释**

每个设计都不是“绝对最好”，而是在复杂度、安全和开发成本之间取平衡。

**专业回答**

第一，选择显式状态机而不是纯 ReAct，因为预约业务需要可测试的状态转移和确认边界。第二，选择单 OperationsAgent 而不是多 Agent，因为当前主要复杂度是同一对话里的状态一致性。第三，普通偏好确认后可写入，但敏感记忆 pending_review，是体验和风险的取舍。第四，不确定排班直接阻断或人工兜底，而不是继续让 LLM 猜。第五，用 Pydantic schema 和 Tool Gateway，而不是只靠 prompt，是为了把业务安全下沉到确定性系统。

### Q100：这个项目目前最大的不足和最担心的线上事故是什么？

**人话解释**

成熟回答不能说“没问题”。要承认当前验证和生产系统之间还有距离。

**专业回答**

当前不足是离线 180+ case 覆盖了核心链路，但仍不能完全代表真实线上长尾表达、并发预约、跨门店资源、接口半失败和 prompt injection 变体。生产化还需要更强的数据库事务、幂等键、线上监控、人工审核后台、隐私脱敏和渠道接入。最担心的事故是误写库，比如未确认创建预约、冲突档期写入、敏感记忆未审核就使用、重复确认导致重复预约。对应控制是 confirmation token、二次排班校验、Tool Gateway、memory review、trace replay 和线上报警。
